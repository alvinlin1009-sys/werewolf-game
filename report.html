01 · Motivation

專案動機

為什麼選擇開發線上版狼人殺？

🎮

經典桌遊數位化

狼人殺是一款需要邏輯推理、心理博弈與社交互動的經典派對遊戲。將其數位化，能讓更多人突破地域限制一起同樂。

⚙️

技術深度挑戰

開發狼人殺需要設計「有限狀態機 (FSM)」、處理即時多人同步 (WebSocket) 和複雜的遊戲邏輯，是一個絕佳的全端工程實作題材。

🚀

解決真實痛點

市面上的狼人殺 APP 需要下載安裝和大量人數，門檻高。本專案希望做到「免安裝、開瀏覽器即玩、AI 機器人自動補數」的輕量化體驗。

🤖 AI 機器人系統

  • 支援任意數量的 AI 機器人補位，解決玩家人數不足的問題。
  • 機器人具備隨機發言（從預設語料庫中選取）、自動投票、自動執行特殊技能（女巫用藥、狼人殺人等）的能力。
  • 狼人陣營中若有真人玩家，AI 機器人的決策以真人玩家的投票為主（多數決），避免 AI 搶奪決策權。

👑 房主權限系統

第一位建立房間的玩家被標記為「房主 (Host)」。透過 hostId 欄位進行後端驗證，只有房主能加入機器人，防止其他玩家惡意干擾遊戲準備階段。

05 · Security Report

資安報告

針對本專案提出的資安見解：漏洞發現、成因分析與解決方案

✅ 已修復漏洞

🔓

漏洞一：全圖透視外掛 (Information Leakage)

嚴重程度：高 (High) · OWASP A02: Cryptographic Failures

⚠️ 成因 (Root Cause)

原先伺服器在每次廣播時，會將完整的 room 物件直接傳給房間內的所有 Socket 連線。這個物件包含了所有玩家的真實身分 (role)、狼人夜間的擊殺目標 (nightKilled)，以及狼人頻道內部的投票紀錄 (wolfVotes)。


懂技術的玩家只需在瀏覽器按下 F12 → Network 面板 → 監聽 WebSocket 封包，即可在不影響遊戲的情況下，實時查閱所有人的底牌，進行作弊。

🛡️ 解決方案 (Solution)

廢除全局廣播機制，改實作「視角過濾器 (Perspective Filter)」函數 getPlayerSpecificState(room, playerId)。


伺服器在傳送狀態更新時，會針對每位玩家的 socket.id 單獨呼叫此函數，確保：

  • 其他玩家的 role 強制覆蓋為 null。
  • 僅狼人能看到隊友身分。
  • 遊戲結束前，nightKilled 等敏感欄位徹底從傳送的封包中刪除。
// 修復後的視角過濾器核心邏輯 (server.js)
const getPlayerSpecificState = (room, playerId) => {
const safeRoom = { ...room };

if (safeRoom.status !== STATUS.GAMEOVER) {
// 刪除所有夜間敏感資料
delete safeRoom.nightKilled;
delete safeRoom.wolfVotes;

// 對每個玩家進行身分遮蔽
safeRoom.players = safeRoom.players.map(p => {
const isSelf = p.id === playerId;
const isWolfTeam = requestingRole === ROLES.WEREWOLF && p.role === ROLES.WEREWOLF;
if (!isSelf && !isWolfTeam) return { ...p, role: null };
return p;
});
}
return safeRoom;
};
🔢

漏洞二：房號暴力破解亂入 (Brute-force Joining)

嚴重程度：中 (Medium) · OWASP A07: Identification and Authentication Failures

⚠️ 成因 (Root Cause)

原先房間代碼僅由 4 碼英數字組合而成，其可能的組合總數約為 36⁴ = 167 萬種。


惡意使用者可輕易撰寫腳本，在短時間內遍歷大量組合，找到正在等待玩家的房間，塞入幽靈帳號或機器人，干擾正常遊戲的準備流程，並對伺服器造成額外負載。

🛡️ 解決方案 (Solution)

將房間代碼的長度由 4 碼提升至 6 碼，組合數從 167 萬種大幅躍升至 36⁶ = 21.7 億種，暴力破解的難度呈指數級增長。


同時，前端的輸入框 maxlength 屬性也同步更新為 6，確保前後端驗證一致。

✅

評估三：跨站腳本攻擊 (XSS) — 已防範

嚴重程度：已緩解 (Mitigated) · OWASP A03: Injection

⚠️ 潛在風險說明

若玩家在聊天室輸入惡意程式碼（如 <script>alert('XSS')</script>），且前端使用 innerHTML 進行渲染，則該程式碼會在所有玩家的瀏覽器上執行，可能用於竊取 Cookie 或發動釣魚攻擊。

🛡️ 現有防護

本專案前端的 addLog() 函數使用 innerText 而非 innerHTML 進行文字渲染。innerText 會自動將所有 HTML 標籤轉義為純文字，從根本上阻斷了 XSS 攻擊的注入途徑。

🔮 未來潛在風險（建議改善）

⚠️ 缺乏玩家身分驗證 (No Auth)

目前玩家僅憑暱稱加入遊戲，無法防止多開帳號或惡意冒名。斷線後因 socket.id 改變，也無法重連到原本座位。建議未來導入 UUID Token 存放於 localStorage，實現斷線重連。

⚠️ 缺乏速率限制 (No Rate Limiting)

目前伺服器對 Socket 事件沒有速率限制，惡意客戶端可在短時間內大量發送 game_action 事件，對伺服器造成 DoS 攻擊。建議使用 socket.io-limiter 等套件進行每秒事件頻率的限制。

⚠️ 缺乏 HTTPS 加密

目前在本機運行時為 HTTP，若部署在不支援 HTTPS 的環境下，傳輸的所有封包（含玩家暱稱與遊戲事件）理論上可被中間人 (Man-in-the-Middle) 攔截。正式部署應搭配 SSL/TLS 憑證（如 Let's Encrypt）。

⚠️ 伺服器端缺乏輸入驗證 (Input Validation)

目前伺服器對 game_action 的 target 參數雖有基本的 isNaN 檢查，但缺乏更嚴格的範圍驗證（如確認目標 seat 值是否在合法玩家座位範圍內），可能被構造的異常輸入利用。

06 · Reflections

心得與反思

開發過程中的挑戰、學習與未來展望

01

狀態同步是最大的挑戰

開發過程中，最困難的部分是確保所有玩家的畫面與伺服器狀態嚴格保持一致。我們曾遭遇計時器 (setTimeout) 被序列化進 Socket 封包，導致 RangeError: Maximum call stack size exceeded 的嚴重崩潰。這讓我學到在傳送資料前必須先進行「安全狀態淨化」，排除無法被 JSON 序列化的物件。

02

「後端永遠不要信任前端」

在實作視角過濾器時，我深刻體會到這個資安原則。遊戲邏輯如果只靠「前端不顯示」來隱藏資訊，在封包層面依然是透明的。真正有效的資訊保護，必須從「伺服器端決定傳什麼」開始，而不是「客戶端決定顯示什麼」。

03

即時遊戲的時序設計

遊戲中有大量倒數計時與「等待所有人完成動作」的邏輯（如：狼人全部投票後立刻結束夜晚）。設計這些流程讓我深刻理解了「非同步程式設計」與「競態條件 (Race Condition)」的實際挑戰，也讓我更加熟悉 Node.js 的事件循環機制。

04

未來展望

目前的系統已具備良好的基礎架構。未來計畫加入：① JWT 認證機制（實現斷線重連）、② WebRTC 語音功能（更接近桌遊體驗）、③ 更多神職角色（白痴、守衛等）、④ 聊天訊息的速率限制，以及 ⑤ 將遊戲正式部署至雲端平台（如 Render 或 Railway）供大眾使用。
